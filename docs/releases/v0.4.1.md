# AgentGuard v0.4.1 — Release notes (draft)

_Status: draft, updated as each phase of the release lands. The final version of this document is the announcement posted on release day._

## TL;DR

v0.4.1 is a behavioral-fix and observability release. The big ticket is the audit-log format bump to `schema_version: 2` (adds a header record, enables rotation, and makes startup replay incremental via a checkpoint file). Everything else is smaller surface: opt-in config flags, better metrics for things that used to be silent, and one policy-loading deprecation warn.

No server or SDK breakage is introduced in this release. The audit-log migration runs automatically and keeps a byte-for-byte backup so downgrade to v0.4.0 is still possible through v0.4.2.

## Who should upgrade

- **Operators running against a growing audit file.** You are the main beneficiary of the schema-2 format: startup replay is now incremental (no more multi-second scans at boot), rotation is built in, and corrupt checkpoints fail loudly instead of silently re-scanning from zero.
- **Operators behind a TLS-terminating reverse proxy.** The new `proxy.tls.terminated_upstream` flag emits `Secure` session cookies unconditionally; useful if your proxy does not forward `X-Forwarded-Proto`.
- **Anyone scraping `/metrics`.** Several previously-invisible failure modes (notification drops, SSE drops, rate-limiter cardinality, body-too-large rejections) now have labeled counters.
- **Policy authors using `conditions.time_window` without `require_prior`.** Your policies still load, but v0.4.1 emits a deprecation warning and v0.5.0 will make it an error. Fix now; don't wait.

## Upgrade path

### From v0.4.0

1. Pull v0.4.1 onto the host.
2. Start the server as usual. On first boot, the `v040_to_v041` migration detects the headerless audit file, rewrites it in place with the new header, and drops a backup at `<audit-log>.v040-backup`.
3. If anything looks wrong, stop the server, run `cp <audit-log>.v040-backup <audit-log>`, and roll back the binary. The backup is retained through v0.4.2; v0.4.3 removes both the migration and the backup.
4. Dry-run option: if you want to preview the migration before enabling it on the running server, run `agentguard migrate --dry-run` against the audit file's path first.

### From v0.3.x

Upgrade to v0.4.0 first, then follow the v0.4.0-to-v0.4.1 steps above. Direct v0.3.x-to-v0.4.1 upgrades are not tested.

### Rolling back

v0.4.1 -> v0.4.0 is supported through v0.4.2: stop the server, restore `<audit-log>.v040-backup` over the live audit log, and replace the binary. The rollback relies on the backup file being present; if you deliberately deleted it, downgrade is not supported and the older v0.4.0 server will refuse to start against a schema-2 file.

## What changed by area

### Audit subsystem (Phase 2.1)

_Largest single change in the release; flagged here because it touches both on-disk format and startup path._

- **schema_version: 2.** First record of every audit file is now a meta-record containing `schema_version`, `created_at`, and (on rotated files) `rotated_from`. Readers at v0.4.1+ transparently handle v1 via migration; older readers see a file they cannot parse and refuse to start.
- **Startup replay is checkpointed.** The replay code maintains `<audit-dir>/.replay-checkpoint` so restart costs `O(new records)` not `O(total history)`. A corrupt checkpoint fails startup with a message pointing at `agentguard migrate --reset-checkpoint` rather than silently re-scanning.
- **Rotation is built in.** Size-triggered, gzip-compressed rotated files, the chain traversed at startup via the `rotated_from` header.
- **Migration (`v040_to_v041`) runs automatically** on the first boot against a schema-1 file. A `.v040-backup` is written alongside. The migration is compiled into the server binary through v0.4.2 and removed in v0.4.3.
- **New observability**: `agentguard_audit_replay_duration_seconds`, `agentguard_audit_replay_entries_total`, `agentguard_audit_rotations_total`, `agentguard_audit_migration_status{from,to,status}`.

### Approval queue (Phase 2.2)

- Eviction is LRU over resolved entries rather than bulk-drop-all-resolved. Under extreme load where no entries are resolved and the queue is full, new approvals receive `503 Service Unavailable` with a `Retry-After` header. Previously the queue would silently lose state. **Heavy approval users may see the new 503 where they did not see failure before**; the server is protecting you from an unbounded queue, and the new metrics make the event visible.

### Session store (Phase 2.3)

- At the hard-coded `MaxSessions = 1024` cap, new logins are rejected with `503` and a clear message. Previously the oldest session was silently evicted, which meant an admin holding a valid tab could be logged out without notice. If you have a dashboard workflow with more than ~1000 concurrent active sessions you will see the new failure; the workaround is to sign out idle tabs.

### Policy watcher (Phase 2.4)

- Moved from 2 s polling to fsnotify, with poll as a fallback. Atomic-replace rewrites (`mv tmp policy.yaml`) now reload within milliseconds. No user-visible contract change.

### HTTP surface (Phase 1)

- `/v1/audit` honors `?limit` (clamped to `[1, 1000]`, default `100`) and `?offset`. The dashboard's `?limit=200` finally does what it says.
- `MaxRequestBodySize` rejections log at WARN and increment `agentguard_request_rejected_total{reason="body_too_large"}`.
- `proxy.tls.terminated_upstream` config flag issues `Secure` cookies unconditionally. Off by default.

### Observability (Phase 3)

- New counters and gauges for notification drops, notification queue depth, notification dispatch latency, SSE drops, SSE subscriber count, rate-limiter bucket count, rate-limiter evictions, approval pending, approval eviction reasons, and deprecation counter usage. See the CHANGELOG for the full list.
- Duration histograms extend past 1 s (new 2.5 s / 5 s / 10 s buckets). Buckets remain non-configurable — changing them invalidates historical data.

### Config surface (Phase 4)

- New top-level keys: `proxy.session.ttl`, `proxy.request.max_body_bytes`, `proxy.audit.default_limit`, `proxy.audit.max_limit`, `notify.dispatch_timeout`. Safe defaults match v0.4.0 behavior. Fully documented in `docs/CONFIG.md`.

### SDK and adapter compat (Phase 5)

- **Python SDK** gains a `fail_mode` parameter on `Guard(...)`, default `"deny"`. No behavior change in v0.4.1; this prepares the v0.5.0 alignment where both SDKs are explicit about the default.
  - Current behavior (pinned down before writing the knob): Python SDK fail-closes to `DENY` on any transport error; TypeScript SDK honors a `failMode` constructor option whose default is `'deny'`. v0.5.0 release notes will tell TS users who relied on implicit `'allow'` to set `failMode: 'allow'` explicitly.
- **MCP adapter** logs a WARN when a client requests a protocol version newer than the pinned `2024-11-05`. Actual negotiation is a v0.5.0 design item tracked by an open issue.

### Deprecations (new this release)

Each is documented with its removal target in `docs/DEPRECATIONS.md`.

- `policy.time_window_without_require_prior` — load-time warning in v0.4.1, error in v0.5.0.
- `audit.migration.v040_to_v041` — auto-runs through v0.4.2, removed in v0.4.3.
- `audit.backup.v040` — `.v040-backup` files retained through v0.4.2, removed with the migration in v0.4.3.

## Things that did not change

- No 1.0 commitment.
- No breaking SDK changes.
- No new auth schemes (Bearer + session cookie only, as before).
- `FileLogger` remains the wired audit backend. `SQLiteLogger` stays dormant in the tree; coexistence design is a v0.5.0 concern.
- `/v1/check` is still intentionally unauthenticated. Gate it behind a reverse proxy if you need network-level auth.

## Known behaviors and gotchas carried forward

Still true as of v0.4.1. Listed here because operators upgrading often re-read this section:

- Setting `--api-key` switches the bind address from `127.0.0.1` to `0.0.0.0`. Unsetting reverts to localhost-only.
- Approval resolution is poll-based from the SDK side; `wait_for_approval` does not block on a server-side signal.
- Permissive-localhost CORS is the default. Set `--allowed-origin` for single-origin strict mode.
- `notify.DroppedEvents` is no longer the only way to see drops — the Prometheus metric from Phase 3 supersedes it for operators, but the atomic remains for Go consumers.
