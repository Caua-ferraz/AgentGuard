# AgentGuard v1.0 Multi-Node — Async-Reconcile Postgres Strategy

> **Status: APPROVED — IMPLEMENTATION PLAN. Approved 2026-07-13.**
> This document defines the design for v1.0 multi-node High Availability (HA) scaling.
> No `.go` files have been modified. Code implementation follows the sequence
> defined in §8 (rollout sequence), with sub-task IDs cross-referenced to TODO.md.
> Governed by the [CLAUDE.md](../CLAUDE.md) engineering contract — in particular
> the **<3 ms p99** latency budget and the **zero-trust `tenant_id`** rule.

## 0. Executive Summary

The v1.0 milestone introduces High Availability across multiple AgentGuard replicas by
sharing state (approvals, rate-limit buckets, cost accumulators) via a shared Postgres
backend, all while preserving the existing <3ms p99 latency guarantee on the firewall's
hot path.

**Key insight:** The hot-path code (`ratelimit.Limiter.Allow`, approval `Lookup`) remains
**completely unchanged**. All database interaction is background-only, following the
**write-behind dual-tier pattern** (in-memory authoritative, async sync to durable store)
already proven in v0.6 by the audit log and session-cost syncer.

---

## 1. Goals & Non-Goals

### Goal
Enable AgentGuard to run as an HA cluster of N replicas with shared mutable state
(approvals, rate-limit buckets, cost accumulators) via a shared Postgres backend,
so that:
- Approvals created on any replica are visible to all others (background visibility delay ~reconcile_interval).
- Rate-limit and cost state is synchronized across replicas for disaster recovery and replay.
- The firewall operates identically whether deployed as single-node or multi-node.

### Non-Goal
A **globally-strict distributed rate limiter** that enforces hard per-second limits across
the entire cluster. This is explicitly out of scope because it would require per-check
coordination with the shared store (a synchronous DB call on the hot path), violating
the <3ms p99 contract. See §3 for the accepted trade-off.

---

## 2. Governing Principle

**Memory stays authoritative per node; Postgres is the shared durable tier; ALL convergence
is background-only.**

The hot-path code is UNCHANGED:
- `ratelimit.Limiter.Allow()` — still reads and updates the in-memory bucket map.
- `proxy.ApprovalQueue.Lookup()` — still reads the in-memory approval map.
- `policy.Engine.Check()` — still evaluates policy against in-memory state.

This preserves the CLAUDE.md invariants:
- ✅ No synchronous DB call on the hot path.
- ✅ Policy and limit decisions operate on in-memory state only.
- ✅ <3ms p99 on `/v1/check` is untouched.

---

## 3. Consistency Model & the Accepted Trade

### Bounded-Overshoot Distributed Rate Limiting

AgentGuard uses **bounded-overshoot** (not globally-strict) rate limiting. With N replicas
and a reconciliation interval of `Δt`, the worst-case burst overshoot is:

```
overshoot ≈ Δt × peak_rate × N
```

For example, with 3 replicas, a reconcile interval of 2s, and a rate limit of 100 req/s per
agent:
```
overshoot ≈ 2s × 100 req/s × 3 = 600 requests
```

This is the **accepted trade** for staying off the hot path and preserving sub-millisecond
latency. A truly global limiter would require per-check coordination with the store,
making it impossible to meet the <3ms p99 budget.

### Approval Visibility Delay

Approvals created on one node become visible to others within `reconcile_interval` (default 2s).
During this window, another replica may not yet know the approval exists. This is bounded by
design: operators set `--reconcile-interval` based on their tolerance for cross-node visibility
delay and reconciliation overhead.

## 4. Architecture & Components

### 4.1 PostgresStore Implementation

**File:** `pkg/store/postgres.go`

A new implementation of the existing `store.Store` interface, mirroring the structure of
`SQLiteStore`, using `pgx` v5 via the standard `database/sql` driver (pure Go, no CGO).

**Interface signature (unchanged from v0.6):**
```go
type Store interface {
    Approval(ctx context.Context, tenantID, approvalID string) (*types.Approval, error)
    ApprovalByRequest(ctx context.Context, tenantID, reqID string) ([]*types.Approval, error)
    UpsertApproval(ctx context.Context, tenantID string, approval *types.Approval) error
    
    RateLimit(ctx context.Context, tenantID, scope string) (*types.RateLimit, error)
    UpsertRateLimit(ctx context.Context, tenantID string, limit *types.RateLimit) error
    
    Cost(ctx context.Context, tenantID, scope string) (*types.Cost, error)
    UpsertCost(ctx context.Context, tenantID string, cost *types.Cost) error
    
    AuditLog(ctx context.Context, tenantID string, offset, limit int) ([]*types.AuditEvent, error)
    AppendAudit(ctx context.Context, tenantID string, event *types.AuditEvent) error
    
    Migrate(ctx context.Context) error
    Ping(ctx context.Context) error
    Close() error
}
```

**Tenant-scoped zero-trust:** Every method takes `tenantID` as a required explicit parameter
(no defaults, no ambient context). Returns `ErrTenantRequired` if tenantID is empty.

**Batch upsert:** Uses `INSERT … ON CONFLICT DO UPDATE` to merge reconciliation deltas
efficiently in a single round-trip.

### 4.2 Backend Dispatcher (`openStore()`)

**File:** `cmd/agentguard/main.go`

**Logic:**
```
if dsn starts with "postgres://" or "postgresql://" → PostgresStore(dsn)
else if --sqlite-path is set → SQLiteStore(--sqlite-path)
else → SQLiteStore(default: "./agentguard.db")
```

- **DSN detection:** Additive; the existing `--store-dsn` flag now accepts both SQLite
  connection strings and Postgres URIs. Empty `--store-dsn` still means in-memory+file audit only.
- **SQLite unchanged:** Existing single-node deployments with `--sqlite-path` or no flag
  continue to work without modification.
- **Pure-Go Postgres:** pgx stdlib driver requires no external C libraries.

### 4.3 Rate & Cost Reconciliation (Background Syncer)

**Files:** `pkg/ratelimit/syncer.go` (new), `pkg/policy/cost_syncer.go` (new)

**Concept:** A background goroutine periodically:
1. **Snapshot** the current in-memory state (rate limits, costs).
2. **Compute deltas** outside the hot-path lock.
3. **Batch upsert** deltas to Postgres via `ApplyDeltas()`.

**Pseudocode:**
```go
func (s *RateLimitSyncer) reconcile(ctx context.Context) {
    ticker := time.NewTicker(s.reconcileInterval)
    defer ticker.Stop()
    
    for {
        select {
        case <-ticker.C:
            // Fast snapshot under read lock
            snapshot := s.limiter.SnapshotBuckets(s.tenantID)
            
            // Compute deltas (no lock held)
            deltas := computeDeltas(snapshot, s.lastSync)
            
            // Small bounded lock: apply locally + queue DB write
            s.limiter.ApplyDeltas(s.tenantID, deltas) // O(deltas), not O(buckets)
            
            // Async: background worker flushes to Postgres
            s.store.UpsertRateLimit(ctx, s.tenantID, deltas...)
            
            s.lastSync = snapshot
        case <-ctx.Done():
            return
        }
    }
}
```

**Key properties:**
- **Hot-path untouched:** `Allow()` still reads the in-memory map; reconciliation does
  not interfere.
- **Bounded lock hold:** The `ApplyDeltas` call is O(number of changed buckets), not
  O(all buckets). Multiple syncers can run in parallel for different tenants without
  lock contention on the entire engine.
- **Async DB flush:** The Postgres write happens on a background worker pool, decoupled
  from the reconciliation loop.
- **Loss tolerance:** If a syncer crashes mid-batch, the in-memory state is still
  authoritative on restart; the next reconciliation will re-sync.

### 4.4 Approval Cross-Node Visibility (Background Hydrator)

**Files:** `pkg/proxy/approval_hydrator.go` (new)

**Concept:** A background goroutine periodically re-hydrates the in-memory approval queue
from Postgres, and optionally uses Postgres `LISTEN`/`NOTIFY` to nudge on approval changes.

**Pseudocode:**
```go
func (h *ApprovalHydrator) hydrate(ctx context.Context) {
    ticker := time.NewTicker(h.reconcileInterval)
    defer ticker.Stop()
    
    for {
        select {
        case <-ticker.C:
            // Fetch all non-expired approvals from Postgres for this tenant
            approvals, err := h.store.Approvals(ctx, h.tenantID, expiredBefore)
            if err != nil {
                // Log, but do not block; keep serving stale in-memory state
                continue
            }
            
            // Rebuild the in-memory queue (RWMutex held for write)
            h.queue.ReplaceApprovals(h.tenantID, approvals)
            
        case <-h.notifyChannel: // LISTEN/NOTIFY nudge
            // Fast path: re-hydrate immediately instead of waiting for ticker
            // (optional, for deployments that want sub-second visibility)
            goto hydrate
            
        case <-ctx.Done():
            return
        }
    }
}
```

**Hot-path behavior (unchanged):**
- `Lookup()` still reads the in-memory map without any database call.
- No latency impact during approval checks.

**Cross-node visibility delay:**
- Bounded by `reconcile_interval` (default 2s) if using only the ticker.
- Can be nudged to sub-second if Postgres `LISTEN`/`NOTIFY` is enabled (opt-in config).

### 4.5 CLI Configuration (Additive Flags)

**File:** `cmd/agentguard/main.go`

New flags (all optional, backward-compatible defaults):

```
--node-id string
    Unique identifier for this firewall instance in the cluster.
    Default: hostname (gethostname)
    
--reconcile-interval duration
    Background reconciliation interval for rate limits, costs, and approvals.
    Default: 2s
    Shorter intervals → faster cross-node visibility but higher DB overhead.
    Longer intervals → lower DB load but more overshoot and stale approvals.
```

**Behavior:**
- If `--store-dsn` is not set or points to SQLite, these flags are ignored.
- If `--store-dsn` is a Postgres URI, reconciliation goroutines start on boot.
- `--node-id` is included in audit logs and metrics for traceability.

## 5. Latency & Allocation Posture

### 5.1 Hot-Path Analysis (Unchanged)

The critical path in `proxy/server.go:handleCheck()` is unchanged:
1. **Approval lookup** — `RLock` map read, in-memory.
2. **Rate-limit Allow** — `Mutex` map op, in-memory.
3. **Policy check** — `RLock` map op + rule walk, in-memory.

**No new allocations, no blocking, no sync DB/network on the hot path.**

### 5.2 Background Reconciliation Overhead

Reconciliation goroutines run on separate OS threads and do not block the request loop:
- **Snapshot latency:** Fast `RLock` + `snapshotBuckets()` → O(changed buckets), typically <1µs.
- **Delta computation:** Outside the lock, negligible.
- **Postgres batch upsert:** Network round-trip ~1–5ms p99 (depending on Postgres proximity),
  but does NOT block the firewall loop.
- **Approval hydration:** Periodic `SELECT` from Postgres, slow but async; if hydration
  falls behind, the in-memory queue is still served.

### 5.3 Lock Contention Analysis

**Risk:** Background `ApplyDeltas()` calls might contend with hot-path `Allow()` on
the same `Limiter.mu`.

**Mitigation:**
- `ApplyDeltas` is O(changed buckets), not O(all buckets).
- Multiple reconciliation goroutines run for different tenants (scoped locks per tenant).
- The lock hold is bounded by design; no waiting for network I/O while holding the lock.

**Verification:** The test suite includes a hard gate — `TestEngineCheck_P99LatencyGate`
(existing, runs at CI time) — which must pass with Postgres reconciliation active.
If p99 exceeds 3ms, the test fails and the build does not proceed.

## 6. Backward Compatibility (v1.0 Lock)

## 7. Testing & CI

### 7.1 Postgres Tests (Env-Gated)

Tests that exercise Postgres interaction are gated by environment variable:

```bash
AGENTGUARD_PG_DSN=postgresql://user:pass@localhost/agentguard_test go test ./pkg/store
```

If `AGENTGUARD_PG_DSN` is unset, Postgres tests are skipped, so the default `go test ./...`
remains deterministic and does not require external dependencies.

### 7.2 Multi-Node Overshoot Test

New test: `TestRateLimitSyncer_BoundedOvershootUnderConcurrency`
- Simulates N replicas with `reconcile_interval` set to a fixed value.
- Sends concurrent requests across replicas that share the same rate-limit bucket.
- Verifies that overshoot stays within predicted bounds.

### 7.3 Store Round-Trip & Tenant Isolation

New tests for `PostgresStore`:
- **`TestPostgresStore_RoundTrip`:** Upsert and fetch Approval, RateLimit, Cost; verify identity.
- **`TestPostgresStore_TenantIsolation`:** Upsert data for tenant A, fetch with tenant B,
  verify `ErrTenantNotFound` or empty result (no cross-tenant leak).
- **`TestPostgresStore_ZeroTrust`:** Call methods with empty tenantID, verify `ErrTenantRequired`.

### 7.4 P99 Latency Gate (Existing, Run With Postgres Active)

The existing `TestEngineCheck_P99LatencyGate` is run as part of CI with:
- Postgres reconciliation active.
- `--reconcile-interval` set to 1s (stress the lock).
- Concurrent approvals and rate-limit updates happening in background.

**Pass criteria:** p99 latency on `Engine.Check` < 3ms. If the test fails, the PR does
not merge.

### 7.5 CI Configuration

**New CI step:**
- Spin up a Postgres 14+ container for the duration of the test run.
- Set `AGENTGUARD_PG_DSN` to the container endpoint.
- Run full test suite (including Postgres-gated tests).
- Tear down container.
- **No testcontainers dependency** — use Docker Compose or a simple `docker run`.

## 8. Rollout Sequence

Implementation is divided into five tracked phases, each with a sub-task ID in TODO.md:

| Phase | ID | Scope | Deliverable |
|-------|----|----|---|
| Foundation | 1-F | `PostgresStore` + backend dispatch | Postgres backend selectable via `--store-dsn`; tests pass with SQLite. Postgres tests skipped (AGENTGUARD_PG_DSN unset). |
| Reconciliation | 1-R | Rate/cost syncer + background loop | `RateLimitSyncer`, `CostSyncer` run; p99 gate passes with Postgres active. |
| Approval Visibility | 1-A | Approval hydrator + LISTEN/NOTIFY opt-in | Background approval re-hydration; cross-node approval visibility within reconcile_interval. |
| Configuration | 1-C | CLI flags + node-id + telemetry | `--node-id`, `--reconcile-interval` flags; audit logs include node-id; metrics include node label. |
| CI & Tests | 1-T | Postgres test container + overshoot bounds | CI runs full suite with Postgres; new multi-node tests pass. |

**Sequential dependency:** Each phase builds on the previous; phases can run in parallel only
within themselves (e.g., multiple implementation files in 1-R can be authored concurrently,
but 1-R tests require 1-F to be complete).

## 9. Operational Considerations

### 9.1 Deployment

**Single-node (existing):**
```bash
agentguard --config config.json
# Uses SQLite in-process (no change)
```

**Multi-node (v1.0+):**
```bash
# Replica 1
agentguard --config config.json --store-dsn "postgresql://user:pass@pghost:5432/agentguard" --node-id "ag-1" --reconcile-interval 2s

# Replica 2
agentguard --config config.json --store-dsn "postgresql://user:pass@pghost:5432/agentguard" --node-id "ag-2" --reconcile-interval 2s
```

### 9.2 Monitoring

**New metrics (Prometheus):**
- `agentguard_syncer_reconcile_latency_ms` (histogram): time to compute and apply deltas.
- `agentguard_store_upsert_latency_ms` (histogram): time to batch upsert to Postgres.
- `agentguard_approval_hydration_lag_seconds` (gauge): age of the youngest approval in the
  in-memory queue vs. Postgres (indicates if hydration is falling behind).
- `agentguard_syncer_delta_count` (counter): number of state changes per reconciliation cycle.

**New audit fields:**
- `node_id`: which firewall instance handled the request.
- `reconciliation_round`: which reconciliation cycle this event was visible in (for debugging).

### 9.3 Postgres Connection Pool

**Recommended settings** (in `config.json` or via env):
```json
{
  "store": {
    "postgres": {
      "dsn": "postgresql://...",
      "max_open_connections": 25,
      "max_idle_connections": 5,
      "connection_max_lifetime": "5m"
    }
  }
}
```

**Rationale:**
- Each reconciliation goroutine (1 per tenant) and approval hydrator (1 per tenant) uses
  one connection during its cycle. With 10 tenants, ~2 reconciliation batches in flight
  = ~20 connections peak.
- 25 max is conservative headroom.
- 5m max lifetime prevents connection drift on long-lived deployments.

### 9.4 Disaster Recovery

**Replica crash → restart:**
1. On restart, the firewall loads its own cached in-memory state (rate limits, approvals).
2. First reconciliation cycle reads the shared Postgres state.
3. Deltas are computed and applied; state converges.
4. No manual intervention needed.

**Postgres unavailable:**
1. Reconciliation goroutines log errors but do not block the firewall.
2. The firewall continues to serve requests using in-memory state.
3. Rate limits and approvals are not persisted; on restart, state is lost.
4. Approvals created during the outage do not sync to other replicas.
5. Once Postgres recovers, the next reconciliation cycle syncs state.

## 10. Future Extensibility (Post-v1.0)

This design is intentionally minimal and extensible:

- **Sharded Postgres:** Replace the single Postgres DSN with a router that shards by tenant_id.
  The `Store` interface does not need to change.
- **Cross-datacenter replication:** Postgres logical replication or streaming can sync state
  across regions; the firewall logic remains identical.
- **Admission control by Postgres lag:** Monitor `agentguard_approval_hydration_lag_seconds`
  and reject new approvals if hydration is stale (not in v1.0; reserved for later).
- **Cryptographic audit trail:** Extend audit schema to include signatures; log to Postgres.
  Wire protocol and policy logic are unaffected.
