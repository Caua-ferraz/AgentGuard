# AgentGuard v0.6 Architecture Plan — Persistent Multi-Tenant State

> **Historical document — superseded.** This was the design plan for the v0.6
> persistence + multi-tenancy milestone; the work described here **shipped**
> (tagged `V0.6.0`, documented under the v0.7.0 release). It is kept for the
> design rationale only — for current behavior see
> [`CLI.md § Persistence & multi-tenancy`](../CLI.md#persistence--multi-tenancy-v06)
> and [`releases/v0.7.0.md`](../releases/v0.7.0.md). Key constraints carried
> into the implementation: the **<3 ms p99** latency budget and the
> **zero-trust `tenant_id`** rule.

## 0. Executive Summary

The most important finding is what is **already done**:

- The **wire protocol is already tenant-aware.** Both gateways POST to
  `/v1/t/<tenant>/check` ([llmproxy/gate.go:457](../../pkg/llmproxy/gate.go#L457),
  [mcpgw/gate.go:359](../../pkg/mcpgw/gate.go#L359)). The central server validates
  the tenant in `withTenant` ([proxy/tenant.go:74](../../pkg/proxy/tenant.go#L74))
  and `Engine.Check` partitions **policy** on it
  ([policy/engine.go:992](../../pkg/policy/engine.go#L992)).
- Both gateways already expose `--tenant-id`
  ([llmproxy/config.go:172](../../pkg/llmproxy/config.go#L172),
  [mcpgw/config.go:118](../../pkg/mcpgw/config.go#L118)).

So v0.6 is **not** a protocol change. The gap is server-side: **every piece of
mutable state except the policy is keyed without a tenant**, and all of it lives
only in memory. The code already flags this in four places with
`TODO(v0.6): shard … by tenantID` ([proxy/tenant.go:16](../../pkg/proxy/tenant.go#L16)).

v0.6 therefore has two coupled jobs:

1. **Tenant-key the existing state** (approval queue, rate-limit buckets,
  cost accumulators, audit log).
2. **Make that state durable** behind a `Store` interface (SQLite single-node,
  Postgres multi-node) **without ever putting a DB call on the hot path** —
  using a write-behind / read-through-cache pattern that the codebase already
  uses three times (buffered audit, session-cost sweeper, policy provider).

### 2.2 Hot-path latency analysis

The "streaming proxy loop" the contract protects is `handleCheck`
([proxy/server.go:461](../../pkg/proxy/server.go#L461)). On the critical path,
in order:

1. Approval-id lookup — `RLock` map read on the approval queue (#2). In-memory.
2. Rate-limit `Allow` — `Mutex` + map op on buckets (#1). In-memory.
3. `Engine.Check` — `RLock` (or `Lock` for `cost` scope) + rule walk (#3, #4). In-memory.
4. Audit write — **already decoupled**: `BufferedAsyncLogger` is on by default
  (`--audit-buffered=true`, [cmd/agentguard/main.go:65](../../cmd/agentguard/main.go#L65)),
  so the request path enqueues and returns; disk I/O happens on worker goroutines.

**Every authoritative read/write on the hot path is an in-memory map operation
under a mutex — sub-microsecond.** The current p99 budget is met *because nothing
on the path does I/O*. The proxy even exposes per-phase timing headers
(`X-AgentGuard-Policy-Ms`, `X-AgentGuard-Audit-Ms`) to prove it
([server.go:693](../../pkg/proxy/server.go#L693)).

**Latency cost of getting this wrong:** a synchronous SQLite call is ~0.1–1 ms
(local file, but serialized behind one writer lock); a Postgres round trip over
the network is **1–5 ms p99 under load** — a single such call in steps 1–3 blows
the entire 3 ms budget on its own, and the cost scope would hold `Engine.mu`
(write lock) across that round trip, serializing *all* checks. This is exactly
what contract rule #1 forbids.

### 2.3 The dual-tier strategy (fast-path cache + async DB sync) — ✅ DECIDED (2026-06-04)

> **DECISION (2026-06-04):** Locked in by the maintainer — v0.6 ships
> **write-behind dual-tier** persistence. This is no longer a recommendation;
> all subsequent design (the syncer, the `Store` interface, the zero-config
> SQLite default in §2.4) builds on it. Write-through / synchronous-DB designs
> are explicitly off the table.

> **Principle:** memory is the source of truth on the request path; the database
> is the source of truth across restarts and nodes. The two reconcile
> asynchronously. This is write-behind caching, not write-through.

```
                      ┌──────────────────────────────────────────────┐
  /v1/t/{tenant}/check│  HOT PATH  (in-memory, <3ms, lock-protected)  │
  ───────────────────▶│  rate-limit ▸ policy ▸ cost ▸ approval lookup │
                      │  reads/writes the in-mem maps ONLY            │
                      └───────────────┬──────────────────────────────┘
                                      │ enqueue mutation (non-blocking,
                                      │ bounded channel, drop/overflow)
                                      ▼
                      ┌──────────────────────────────────────────────┐
  COLD PATH (async)   │  Syncer goroutine(s) — batched write-behind   │
                      │  • flush dirty buckets/costs every N ms       │
                      │  • append audit batches (already exists)      │
                      │  • GC resolved approvals / stale costs        │
                      └───────────────┬──────────────────────────────┘
                                      ▼
                      ┌──────────────────────────────────────────────┐
                      │  Store  (SQLite / Postgres) — durable, multi- │
                      │  tenant, every method requires tenant_id      │
                      └──────────────────────────────────────────────┘
                                      ▲
                  boot ────────────────┘  hydrate in-mem maps from Store
                  (LoadBuckets / LoadCosts / ListPendingApprovals)
```

**Mechanics by structure:**

- **Rate-limit buckets (#1):** stay authoritative in memory. A syncer snapshots
  dirty buckets to the Store on a ticker (default ~1 s) and on graceful
  shutdown. On boot, `LoadBuckets(tenant)` rehydrates so a restart doesn't reset
  every limiter to full. Single-node SQLite: snapshot is sufficient. Multi-node
  Postgres: buckets become **node-local with periodic reconciliation** (an
  operator who needs a globally-exact distributed limiter opts into a strict
  mode that accepts the latency — see §4 `RateLimitStore` note). This honors
  contract rule #2 ("async syncs only for rate-limits and cost-accumulators").
- **Cost accumulators (#3):** identical pattern. The atomic check-and-reserve
  ([engine.go:1334](../../pkg/policy/engine.go#L1334)) stays in memory under
  `Engine.mu`; deltas are flushed write-behind. The existing `SweepSessionCosts`
  sweeper ([server.go:419](../../pkg/proxy/server.go#L419)) becomes the natural
  home to also flush+GC against the Store.
- **Approval queue (#2):** the *lookup* on retry stays in memory (it's on the
  hot path). `Add`/`Resolve` additionally enqueue a write-behind upsert so
  approvals survive a restart (today a restart loses every pending approval).
  `ListPendingApprovals(tenant)` rehydrates on boot.
- **Audit (#6):** already write-behind via `BufferedAsyncLogger`. v0.6 swaps the
  sink from `FileLogger` to a Store-backed `AuditSink` (the `SQLiteLogger`
  already exists, just unwired — `sqlite_logger.go:10`),
  and adds the `tenant_id` column.

**Existing precedents we are copying (not inventing):**
`BufferedAsyncLogger` (bounded queue + workers + disk overflow,
[buffered.go](../../pkg/audit/buffered.go)); the session-cost sweeper goroutine;
the `PolicyProvider` read-through abstraction. The dual-tier design is the same
shape applied to three more structures.

**Failure posture:** if the Store is down, the hot path is unaffected (memory
still authoritative). Write-behind queues bound and overflow/drop with a metric,
exactly like the notify dispatcher and buffered audit do today. Durability
degrades; availability and latency do not. This must be a stated, tested
invariant.

### 2.4 Zero-config local-first default (SQLite)

The v0.6 default is **zero-config durability**: `agentguard serve` with no
database settings yields a persistent, single-node deployment automatically —
no Postgres, no DSN, no extra flags. Boot sequence (each step tags where the
hosted build diverges):

1. **DB detection.** If no external DB URL is supplied (`--store-dsn` empty and
  `$AGENTGUARD_STORE_DSN` unset), AgentGuard selects the embedded SQLite
  backend.
  **`(Changed in cloud version)`** — the hosted build always supplies a
  Postgres DSN, so detection resolves to `PostgresStore` and the local-disk
  branch is never taken.

2. **Auto-provision.** It creates `agentguard.db` in the working directory
  (overridable via `--data-dir`), runs `Migrate`, and sets
  `PRAGMA journal_mode=WAL` so the dashboard can read concurrently with the
  syncer's writes without blocking.
  **`(Changed in cloud version)`** — no local file is created; schema migration
  runs against managed Postgres and must be **leader-gated** so concurrent
  nodes don't race the DDL on startup.

3. **In-memory fast path.** The rate limiters
  ([ratelimit/ratelimit.go:19](../../pkg/ratelimit/ratelimit.go#L19)) and cost
  trackers ([engine.go:697](../../pkg/policy/engine.go#L697)) come up empty or
  hydrated from SQLite (`LoadBuckets` / `LoadCosts`) and serve `/v1/check`
  **entirely from memory** — this is what keeps the <3 ms p99 budget intact.
  **`(Changed in cloud version)`** — per-node memory stays, but buckets and cost
  accumulators become **node-local with cross-node reconciliation** (or opt-in
  strict `ReserveTokens`); a globally exact limiter is a deliberate latency
  trade-off, never the default. See §6 Q3.

4. **Background syncer.** One goroutine flushes dirty memory state **and** audit
  batches into `agentguard.db` on a ticker with a **hard 1 s floor** — it never
  syncs more often than once per second, bounding write amplification and WAL
  churn — and also flushes on graceful shutdown. It is **off the hot path** by
  construction (contract rule #1).
  **`(Changed in cloud version)`** — the syncer targets Postgres over the
  network; batch sizes and the flush floor are tuned for RTT and multiple
  concurrent writers (the floor may be raised above 1 s), and conflict
  resolution uses upserts keyed by `(tenant_id, …)`.

5. **Dashboard reads from SQLite directly.** The dashboard data endpoints
  (`/api/stats`, `/api/pending`, `/v1/audit` — [server.go:355](../../pkg/proxy/server.go#L355))
  read the local `agentguard.db` file directly; WAL makes these reads
  non-blocking against the syncer.
  **`(Changed in cloud version)`** — **largest divergence.** There is no shared
  local file across nodes, so the dashboard MUST read **through the `Store` /
  HTTP API or a Postgres read replica**, never a file handle. Any code that
  opens `agentguard.db` by path has to route through the `Store` abstraction in
  the hosted build.

Net effect: a single binary, a single file, sub-3 ms checks, and durable state
across restarts — with a clean seam (`Store`) so the same code scales out to
Postgres by swapping the implementation, not the call sites.

### 3.2 Policy YAML structs

Defined in [policy/engine.go:44-226](../../pkg/policy/engine.go#L44) (`Policy`,
`RuleSet`, `Rule`, `RateLimitCfg`, `CostLimits`, `AgentCfg`, `NotificationCfg`,
`ProxyCfg`, …).

- **`Policy` has no `tenant_id` — and should not.** Tenant identity is *external*
  to the document: the `PolicyProvider` maps `tenantID → *Policy`
  ([provider.go:50](../../pkg/policy/provider.go#L50)). A DB-backed multi-tenant
  provider stores one policy row per tenant and returns the right one from
  `Get(tenantID)`. Embedding `tenant_id` inside the YAML would duplicate the key
  and invite drift. **Keep tenant out of the policy body.**
- The multi-tenant provider is the clean injection seam: implement
  `DBPolicyProvider` satisfying the existing interface; `Engine` and `Server`
  need **zero** changes to consume multiple tenants for policy
  (`PolicyForTenant` already forwards, [engine.go:913](../../pkg/policy/engine.go#L913)).
- One real decision: today `Engine` caches **only** the `local` policy and
  `Watch`es only `local`. Multi-tenant means either (a) cache-per-tenant with
  per-tenant `Watch`, or (b) keep `Check` going through the provider for
  non-local tenants (slower, but the provider can have its own in-mem cache).
  Recommend **(a)** with a small `map[tenant]*Policy` LRU + lazy `Watch`.

### 3.3 Audit log schema (JSONL + SQLite) — the real v1.0 break point

- **JSONL `audit.Entry`** ([audit/logger.go:53](../../pkg/audit/logger.go#L53)) has
  fields `Timestamp, SessionID, AgentID, Request, Result, DurationMs, Transport`
  — **no `TenantID`.**
- **SQLite `audit_entries`** (`sqlite_logger.go:46`)
  has no `tenant_id` column and no index on it.
- **`QueryFilter`** ([logger.go:98](../../pkg/audit/logger.go#L98)) cannot filter by
  tenant; `/v1/t/{tenant}/audit` therefore returns **all tenants' entries today**
  — a tenant-isolation defect that ships the moment a second tenant exists.

This is the schema that v1.0 promises to keep stable. Adding `tenant_id` is
**additive and backward-compatible** if done now:

- `Transport` set the precedent: it was added without bumping the audit
  `schema_version` (still `2`) because it is `omitempty` and readers default it
  ([logger.go:44-84](../../pkg/audit/logger.go#L44)). `tenant_id` follows the same
  rule: `omitempty`, default `"local"` on read.
- Byte-identity tests (`at_byte_identity_e2e_test.go`) stay green for the
  single-tenant `local` case because `omitempty` + default `"local"` keeps the
  serialized form unchanged.

## Phase 4 — The `Store` Interface Draft

Proposed package: `pkg/store`. The `Store` is the **durable, cold-path tier**
from §2.3. It is **never called synchronously from `handleCheck`**; the syncer
goroutines and boot-time hydration call it. Two implementations: `SQLiteStore`
(single-node, the zero-config default of §2.4, reuses the `modernc.org/sqlite`
pure-Go driver the `SQLiteLogger` already documents) and `PostgresStore`
(multi-node) **`(Changed in cloud version)`** — the hosted build wires
`PostgresStore`; `SQLiteStore` is never instantiated server-side.

Design rules baked into the interface:

- **Zero-trust:** every method takes `ctx context.Context` and a non-empty
  `tenantID`; implementations reject `""` with `ErrTenantRequired` (contract #3).
- **Batch-friendly:** mutating methods accept slices so the write-behind syncer
  flushes in batches, not per-event.
- **Hydration-friendly:** every store has a `Load*` method to rebuild the
  in-memory map on boot.
- **GC-friendly:** every accreting store has a `*Before(cutoff)` reaper, mirroring
  `SweepSessionCosts` and the approval LRU.

```go
// Package store is AgentGuard's durable, multi-tenant persistence tier.
//
// CONTRACT (see docs/v0.6-ARCHITECTURE-PLAN.md and CLAUDE.md):
//   - Implementations MUST NOT be called on the streaming proxy hot path.
//     The in-memory maps in pkg/ratelimit, pkg/proxy (ApprovalQueue) and
//     pkg/policy (Engine.sessionCosts) remain authoritative for /v1/check;
//     the Store is reconciled asynchronously (write-behind) and read only
//     at boot (hydration) or from operator/query paths.
//   - Every method requires a non-empty tenantID and rejects "" with
//     ErrTenantRequired (zero-trust). "local" is the default single-tenant id.
//   - Mutating methods are batch-shaped so the syncer flushes efficiently.
package store

import (
  "context"
  "errors"
  "time"

  "github.com/Caua-ferraz/AgentGuard/pkg/audit"
  "github.com/Caua-ferraz/AgentGuard/pkg/policy"
)

// ErrTenantRequired is returned by any Store method called with an empty
// tenantID. Enforces the zero-trust interface rule at the lowest layer so a
// caller that forgets to thread the tenant fails loudly rather than silently
// reading/writing a cross-tenant or "global" partition.
var ErrTenantRequired = errors.New("store: tenant_id is required")

// Store is the top-level durable persistence abstraction. SQLiteStore and
// PostgresStore satisfy it. Composed of focused sub-interfaces so call sites
// (the rate-limit syncer, the cost syncer, the approval syncer, the audit
// sink) depend only on what they use.
type Store interface {
  ApprovalStore
  RateLimitStore
  CostStore
  AuditSink

  // Migrate creates or upgrades the schema for all sub-stores. Idempotent
  // (CREATE TABLE IF NOT EXISTS / additive ALTERs); safe to run on every boot.
  Migrate(ctx context.Context) error
  // Ping verifies connectivity for /v1/health without touching tenant data.
  Ping(ctx context.Context) error
  // Close releases pooled connections. Idempotent.
  Close() error
}

// ---------------------------------------------------------------------------
// Approval queue — CRUD. Mirrors pkg/proxy.PendingAction. Backs durable,
// restart-surviving approvals and the dashboard's per-tenant pending list.
// ---------------------------------------------------------------------------

// ApprovalRecord is the persisted form of pkg/proxy.PendingAction plus its
// owning tenant. Kept in pkg/store (not reusing the proxy type) to avoid a
// pkg/store → pkg/proxy import cycle; the proxy maps between the two.
type ApprovalRecord struct {
  TenantID  string
  ID        string // "ap_..." id minted by the proxy
  Request   policy.ActionRequest
  Result    policy.CheckResult
  CreatedAt time.Time
  Resolved  bool
  Decision  string // "ALLOW" | "DENY" | "" while pending
  ResolvedAt time.Time
}

type ApprovalStore interface {
  // UpsertApprovals persists new or updated approvals (write-behind from
  // ApprovalQueue.Add / .Resolve). Batched. Idempotent on (tenant_id, id).
  UpsertApprovals(ctx context.Context, tenantID string, recs []ApprovalRecord) error

  // GetApproval fetches one approval by id, scoped to the tenant. ok=false
  // when absent. Used for cross-node lookups and operator tooling — NOT the
  // hot-path retry lookup (that stays in memory).
  GetApproval(ctx context.Context, tenantID, id string) (rec ApprovalRecord, ok bool, err error)

  // ListPendingApprovals hydrates the in-memory queue on boot: returns
  // unresolved approvals for the tenant, newest first, capped by limit
  // (0 = no cap). Resolved entries are intentionally excluded.
  ListPendingApprovals(ctx context.Context, tenantID string, limit int) ([]ApprovalRecord, error)

  // DeleteResolvedApprovalsBefore reaps resolved approvals older than cutoff
  // (durable analogue of the in-memory LRU eviction). Returns rows deleted.
  DeleteResolvedApprovalsBefore(ctx context.Context, tenantID string, cutoff time.Time) (int, error)
}

// ---------------------------------------------------------------------------
// Rate-limit buckets — increment/check + snapshot/restore.
//
// HOT-PATH NOTE: the authoritative token-bucket check stays in
// pkg/ratelimit.Limiter (in memory). Snapshot/Load implement the write-behind
// persistence + boot hydration of §2.3. ReserveTokens is the OPTIONAL
// strongly-consistent cross-node path: a PostgresStore can implement it as a
// single atomic UPDATE…RETURNING for operators who need a globally exact
// distributed limiter and explicitly accept the per-call latency (it is NEVER
// wired into handleCheck by default).
// ---------------------------------------------------------------------------

// BucketState is one token-bucket's persisted state, keyed by (tenant, key)
// where key is the limiter's "scope:agent_id" string.
type BucketState struct {
  TenantID   string
  Key        string // "scope:agent_id"
  Tokens     int
  Max        int
  Window     time.Duration
  LastRefill time.Time
}

type RateLimitStore interface {
  // SnapshotBuckets persists dirty buckets for a tenant (write-behind, batched,
  // ticker-driven + on shutdown). Upsert on (tenant_id, key).
  SnapshotBuckets(ctx context.Context, tenantID string, buckets []BucketState) error

  // LoadBuckets hydrates a tenant's buckets on boot so a restart does not
  // reset every limiter to full.
  LoadBuckets(ctx context.Context, tenantID string) ([]BucketState, error)

  // ReserveTokens is the OPTIONAL atomic cross-node reservation (see note
  // above). allowed=false means the bucket is exhausted. Off the hot path;
  // used only in an opt-in strict-distributed mode. SQLiteStore may return
  // ErrUnsupported.
  ReserveTokens(ctx context.Context, tenantID, key string, n, max int, window time.Duration) (allowed bool, remaining int, err error)
}

// ---------------------------------------------------------------------------
// Cost accumulators — increment/check + snapshot/restore. Same dual-tier shape
// as rate limits. Authoritative reserve stays in Engine.checkCost under
// Engine.mu; these persist and hydrate it.
// ---------------------------------------------------------------------------

type CostStore interface {
  // AddCostDeltas applies write-behind cost reservations/refunds for a tenant.
  // Each delta is (sessionID, amount); amount may be negative (refund).
  // Batched; the new persisted total is implementation-internal.
  AddCostDeltas(ctx context.Context, tenantID string, deltas []CostDelta) error

  // GetSessionCost returns the durable accumulated cost for one session
  // (operator/query path and cross-node reconciliation; not the hot path).
  GetSessionCost(ctx context.Context, tenantID, sessionID string) (float64, error)

  // LoadCosts hydrates Engine.sessionCosts for a tenant on boot.
  LoadCosts(ctx context.Context, tenantID string) (map[string]float64, error)

  // DeleteCostsBefore reaps session-cost rows whose last update predates
  // cutoff — durable analogue of Engine.SweepSessionCosts. Returns rows deleted.
  DeleteCostsBefore(ctx context.Context, tenantID string, cutoff time.Time) (int, error)
}

// CostDelta is one session-cost mutation queued for write-behind flush.
type CostDelta struct {
  SessionID string
  Amount    float64 // negative = refund
  At        time.Time
}

// ---------------------------------------------------------------------------
// Audit persistence — batched append + indexed query. Replaces the unwired
// SQLiteLogger with a tenant-aware, batch-shaped sink fed by the existing
// BufferedAsyncLogger worker pool. AuditSink is adaptable to the existing
// audit.Logger interface so cmd/agentguard can wire it in place of FileLogger.
// ---------------------------------------------------------------------------

type AuditSink interface {
  // AppendAudit persists a batch of entries for a tenant (called by the
  // buffered async workers, never inline on /v1/check).
  AppendAudit(ctx context.Context, tenantID string, entries []audit.Entry) error

  // QueryAudit returns entries matching filter, scoped to the tenant. Backs
  // GET /v1/t/{tenant}/audit. The tenant scoping here is what closes the
  // cross-tenant audit leak described in §3.3.
  QueryAudit(ctx context.Context, tenantID string, filter audit.QueryFilter) ([]audit.Entry, error)
}
```

### 4.1 Why these signatures

- **Sub-interfaces over one fat interface:** the rate-limit syncer should not be
  able to touch audit rows. Smaller seams = smaller blast radius and trivial
  test fakes. `Store` still composes them for the single wiring point in
  `cmd/agentguard`.
- **Slices everywhere on writes:** the write-behind syncer accumulates dirty
  entries between ticks and flushes one batch — one transaction, not N.
- **`Load*` on every store:** boot hydration is a first-class operation, not an
  afterthought; it's how a restart doesn't lose pending approvals or reset
  limiters.
- **`ReserveTokens` is optional and flagged:** it's the only method that *could*
  be synchronous-distributed, and it's explicitly fenced off from the hot path
  so the contract can't be violated by accident. SQLite returns `ErrUnsupported`.
- **No `tenant_id` in `ActionRequest`:** the Store carries tenant in its own
  columns; the wire body stays v1-frozen.

## 5. Backward-Compatibility & Migration Notes

- **Single-tenant deployments are unchanged.** Default tenant `"local"`,
  `omitempty` JSON, `DEFAULT 'local'` columns ⇒ existing audit files, fixtures,
  byte-identity tests, and SDK callers behave identically.
- **Audit schema_version stays `2`** (additive column, same precedent as the
  `Transport` field). A one-time `ALTER TABLE audit_entries ADD COLUMN tenant_id`
  runs in `Migrate`; the JSONL reader defaults missing `tenant_id` to `"local"`.
- **No wire-protocol bump.** `schema_version: v1` for `/v1/check` request/response
  is untouched; tenant remains a URL-path concern.
- **SQLite is the zero-config default (§2.4).** No DSN ⇒ AgentGuard
  auto-provisions `agentguard.db` (WAL) and persists state + audit there; the
  legacy `--audit-backend=file` JSONL path stays available for operators who
  want it, and `--store-dsn` selects Postgres. The multi-tenant *correctness*
  fixes (tenant-keying #1–#7) are independent of which backend is wired, so they
  can land first. **`(Changed in cloud version)`** — the default is bypassed
  entirely; the hosted build always runs Postgres.

## 6. Decisions & Open Questions

1. ✅ **RESOLVED (2026-06-04) — Server tenancy config model.** Chose **(a):
  `--store-dsn` owns it.** Empty `--store-dsn` ⇒ auto-SQLite (`agentguard.db`,
  tenant `local`); a DSN ⇒ Postgres, and a DB-backed `PolicyProvider`/`Store`
  owns the tenant list as **rows** (tenants added/removed at runtime, no
  restart). This is the minimal flag surface to freeze at v1.0. Rejected:
  (b) a separate `tenants.yaml` config file (second config surface; edits
  need reload), (c) one-process-per-tenant (pushes fan-out to deployment,
  undercuts the multi-tenant goal).
2. **SQLite vs Postgres ordering.** Ship `SQLiteStore` first (single-node,
  reuses the already-documented `modernc.org/sqlite` driver) and `PostgresStore`
  second? Recommendation: **yes**.
3. **Cross-node rate limiting.** Accept node-local buckets + async reconcile as
  the v0.6 default (fast, slightly over-permissive across nodes), with strict
  `ReserveTokens` as opt-in? Recommendation: **yes** — never put the strict
  path on the hot path by default.
4. **Engine multi-tenant policy caching.** Per-tenant cache + lazy `Watch` (a),
  vs. provider-cached pass-through (b)? Recommendation: **(a)**.
